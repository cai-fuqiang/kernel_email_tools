# PLAN-38002: AI 分析能力诊断实验

## 背景

PLAN-38000 提出了核心困惑：想让 AI 协助分析内核邮件，但效果不好。
PLAN-38001 给出了完整的架构重定向方案（EvidenceRef、ContextPack、Draft Review），
但它跳过了最关键的一步——**没有诊断 AI 分析为什么效果不好**。

在投入大量工程资源做架构重构之前，需要先回答一个根本问题：

> AI 在获得足够好的上下文后，到底能不能产出有审核价值的内核知识分析？

如果答案是"能"，那么 ContextPack、EvidenceRef 等抽象就是正确的下一步。
如果答案是"不能"，那么架构重构再多也无法解决核心问题，需要换思路。

## 诊断框架

当前 AI 分析效果不好的可能原因，从最可能到最不可能：

| 序号 | 可能原因 | 严重程度 | 解决方案 |
|------|----------|----------|----------|
| 1 | 上下文不足 — 线程太长，相关补丁/提交/代码没被正确打包进 prompt | 最可能 | ContextPack 是正确的方向 |
| 2 | 检索不准 — 找不到与当前话题相关的历史讨论和代码 | 很可能 | 改进检索策略，或人工构建上下文先验证 |
| 3 | Prompt 设计问题 — 任务定义不清晰，AI 不知道该产出什么 | 可能 | 改进 prompt 模板和输出约束 |
| 4 | 模型能力不足 — 内核对模型来说太专业，幻觉严重 | 有可能 | 缩小任务范围，或接受 AI 只做辅助 |
| 5 | 任务本身太难 — 某些内核话题确实需要极深的背景知识 | 对特定话题成立 | 识别 AI 能做和不能做的边界 |

## 诊断实验设计

### 实验目标

验证：给定人工精心构造的完整上下文，AI 能否产出一份有审核价值的内核知识分析。

### 实验步骤

```
1. 选择一个你已经充分理解的邮件线程
   - 标准：你自己能写出正确答案
   - 有明确的结论或设计决策
   - 涉及代码、补丁、讨论

2. 手工构造"理想上下文"
   - 完整的线程消息（不是摘要，是原文）
   - 相关的补丁代码片段
   - 相关的内核源码上下文
   - 相关的前置讨论或引用
   - 不要依赖自动检索

3. 定义明确的分析任务
   - 这个线程讨论的问题是什么？
   - 提出了哪些方案？各自的 tradeoff 是什么？
   - 最终采纳了什么方案？为什么？
   - 有哪些反对意见？为什么被 reject？
   - 对后续版本有什么影响？

4. 让 AI 产出分析草稿

5. 对比你的已知答案，评估：
   - 事实准确度（有没有编造）
   - 完整性（有没有遗漏关键讨论）
   - 洞察深度（有没有超出表面总结的洞察）
   - 审核成本（你需要花多少时间验证这份草稿）
```

### 成功标准

| 维度 | 及格线 | 优秀线 |
|------|--------|--------|
| 事实准确度 | >80% 的事实陈述正确 | >95%，且错误都是次要细节 |
| 完整性 | 覆盖了主要的讨论分支 | 覆盖了边缘讨论和隐含假设 |
| 洞察深度 | 正确总结了结论 | 指出了讨论中的未解决问题或后续影响 |
| 审核成本 | 审核比从头分析节省 >30% 时间 | 审核比从头分析节省 >70% 时间 |

### 判定逻辑

```
if 及格线全部达到:
    → ContextPack + EvidenceRef 是正确的方向，值得投入架构重构
    → 重点关注"审核成本"这个指标，它决定了 Draft Review 流程是否可持续

elif 事实准确度 < 80%:
    → 模型能力和/或 prompt 设计是核心瓶颈
    → 需要缩小 AI 的任务范围（比如只做摘要，不做分析）
    → 或者探索更好的 prompt 策略、few-shot 示例

elif 完整性 < 及格线:
    → 上下文打包策略需要改进
    → 可能需要多次迭代（第一轮产出草稿 → 发现缺口 → 补充上下文 → 第二轮完善）

elif 审核成本太高:
    → Draft Review 流程在当前阶段不可行
    → 考虑降低 AI 输出形式（从"知识条目"降为"高亮摘要"或"关联线索"）
    → 等模型能力提升后再提升形式
```

## 对 PLAN-38001 的修正建议

无论实验结果如何，PLAN-38001 中有两个观点需要调整：

### 1. 先跑 PoC，再建抽象

PLAN-38001 的 Phase 顺序是：命名 → EvidenceRef → ContextPack → Draft Bundle → Survey PoC。
**应该在 Phase 1 之后直接跳到 Phase 5 的 PoC**，用最糙的方式（手工构造 ContextPack）
验证核心链路是否跑得通。跑通了再回头做 EvidenceRef 和 ContextPack 的工程化。

### 2. 不要低估审核带宽瓶颈

PLAN-38001 说 "AI output must flow through Draft Review before becoming Knowledge"。
这个原则是对的，但需要加上一个现实约束：

> 如果一个知识条目需要的审核时间超过从头研究的时间，Draft Review 就是失败的。

这意味着：
- 不是所有 AI 产出都值得审核——需要质量门槛
- 需要建立"快速驳回"机制——一眼就能看出 AI 在胡说的草稿不要花时间
- 审核 UI 需要突出显示 AI 的"信心"和"证据链"，帮助审核者快速定位风险点

## 原则

1. **用数据验证假设，不要用架构替代实验。**
2. **如果 AI 质量不够好，简化流程比增加抽象更有效。**
3. **一个人维护的知识库，审核带宽是真正的瓶颈。**
4. **PoC 永远应该在架构设计之前。**
