# code-survey 方案对比分析

## 项目简介

[code-survey](https://github.com/eunomia-bpf/code-survey) 是一个用 LLM 系统分析大规模代码库（主要是 Linux 内核）的原型项目。其核心洞见是：**把 LLM 当作问卷填答者，而非聊天助手或专家系统。**

论文: https://arxiv.org/abs/2410.01837

## 核心方法论

```
人类专家设计问卷（选择题为主）
  → LLM 批量跑问卷（从预定义选项中选答案）
  → 统计学验证（多次运行、一致性检查）
  → 产出结构化 CSV 供定量分析
```

### 关键设计决策

#### 1. AI 做选择题，不做生成题

问卷以 `single_choice` 为主，每个选项是预定义标签。LLM 的工作是"选一个"，不是"写一段"。"I'm not sure" 是合法选项，允许 AI 承认不确定性。

```yaml
# 示例：commit 分类问卷
- id: commit_classification
  type: single_choice
  question: "这个 commit 的主要类型是什么？"
  choices:
    - A bug fix
    - A new feature
    - A performance optimization
    - A cleanup or refactoring
    - A documentation change
    - A security fix
    - ...
    - I'm not sure
```

**为什么有效**：限制了 LLM 的创造性空间，答案可量化、可比较、可统计。

#### 2. 统计验证，不逐条人工审核

- 同一条数据跑多次（不同模型或多次运行）
- 用 inter-rater reliability 检查一致性
- 不一致的结果标记为可疑，人只抽查异常值
- 用方差分析检查答案分布是否合理

**和本项目的关键差异**：PLAN-38001 设计的是 "AI draft → 人工逐条审核 → durable knowledge"，code-survey 是 "AI 批量作答 → 统计验证 → 人只抽查异常"。后者的审核带宽友好得多。

#### 3. 先 commit 后邮件

他们的 eBPF 数据集优先级：
- 680+ 人类专家标注的 commit（ground truth）
- 12,000+ LLM 问卷过的 commit（结构化 CSV）← 已完成
- 150,000+ 邮件问卷 ← 仍在 TODO

**原因**：commit message 短（平均几十到几百字），结构化度高，比邮件容易处理得多。

#### 4. 不做代码↔邮件链接

Commit 问卷和邮件问卷独立运行，各自产出结构化数据。在分析阶段可以 join（比如"这个 feature 的 commit 和邮件讨论"），但系统不维护硬链接。

#### 5. 明确反对 RAG

README 原话："No Chatbots or RAG! Stop the ineffective AI!"。他们的论点是：RAG 只能找相似片段，无法替代对大规模非结构化数据的系统性分析与结构化。

## 和当前项目对比

| 维度 | 当前项目 | code-survey |
|------|----------|-------------|
| AI 角色 | 聊天助手、研究代理、草稿生成 | 问卷填答者 |
| 输出形式 | 自由文本、Knowledge Draft | 选择题答案、结构化 CSV |
| 验证方式 | 人工逐条审核 | 统计一致性检验 + 人抽查异常 |
| 核心交互 | 对话式、探索式 | 批量跑问卷 |
| 对 RAG 的态度 | 核心依赖 | 明确反对 |
| 邮件处理 | 实时检索、上下文打包 | 和 commit 一样的问卷批处理 |
| 知识沉淀 | Knowledge Entity + Evidence + Relation | CSV 数据集 + 分析报告 |
| 用户角色 | 阅读、搜索、review draft | 设计问卷、解读统计结果 |

## 不同方法的适用场景

### code-survey 模式适合

- **大规模分类任务**：把 10000 个 commit 分成 10 个类别
- **趋势分析**：eBPF 子系统在 2020-2024 年间，bug fix 的比例是上升还是下降？
- **特征提取**：从所有 commit 中提取关键词、模块名、贡献者角色
- **假设验证**：人工提出假设（"CFS 的 bug 密度在 2012 年后明显下降"），用问卷数据验证

### 当前项目（证据驱动）模式适合

- **深度理解单个话题**：这个 patch 讨论为什么选了方案 A 而不是方案 B？
- **跨源证据聚合**：代码 + commit + 邮件 + 手册 + LWN 综合解释一个设计决策
- **知识积累**：用户 3 个月前查过这个话题，今天可以快速回顾
- **交互式探索**："这段代码是什么意思？" → "相关的讨论有哪些？" → "为什么这么设计？"

## 对本项目的启示

### 可以借鉴的

1. **选择题优于自由文本。** 当 AI 的任务从"写一篇分析"变成"在 10 个选项里选一个"，准确率大幅提升。ContextPack 里的某些分析任务可以从"生成摘要"改为"选择分类"。

2. **统计验证是对人工审核的有力补充。** 不要所有 AI 产出都走人工审核。低风险任务（如 commit 分类）用多次运行 + 一致性检查。高风险任务（如知识实体创建）保留人工审核。

3. **先 commit 后邮件。** 在把邮件分析流程搞复杂之前，先在 commit 上验证方法和质量基线。Commit 更短、更结构化，是更好的实验对象。

4. **允许 AI 说"我不确定"。** 强制 AI 对每个问题都给出答案会导致编造。在设计 prompt 和问卷时，始终提供不确定的出口。

### 不能照搬的

1. **不需要放弃 RAG 和对话。** code-survey 的问卷模式适合大规模分类和趋势分析。本项目的核心场景（深度理解单个话题）确实需要检索和交互。两者解决的是不同层次的问题：问卷解决"宏观"，RAG 解决"微观"。

2. **不需要变成 CSV 工具。** 问卷产出的结构化 CSV 适合统计分析，但不适合作为理解单个设计决策的证据。Knowledge Entity + Evidence 仍然是更好的知识沉淀形式。

3. **不需要独立设计。** code-survey 的问卷设计是核心 IP。但实际上，AI 辅助的"自由文本分析"和 AI 批量填问卷可以作为同一系统的两种模式，针对不同任务使用。

## 一个可能的综合方案

```
宏观层（问卷模式）：
  对大量 commit/邮件 跑分类问卷 → 结构化数据 → 统计洞察
  用途：识别趋势、发现异常、定位"值得深挖"的话题

微观层（证据驱动模式）：
  对单个话题 检索相关 commit/邮件/代码 → ContextPack → AI 分析草稿 → 人工审核 → 知识实体
  用途：深度理解设计决策、积累可信知识

两者关系：
  宏观（问卷）发现"eBPF verifier 相关的 bug fix 在 2023 年激增"
  → 微观（证据驱动）调查"为什么激增？具体是什么问题？怎么解决的？"
  → 产出 Knowledge Entity："eBPF verifier 在 6.5 的重写动机和影响"
```

但这意味着两套系统，复杂度翻倍。在项目初期可能不值得同时搞。

## 建议

短期（当前阶段）：
1. 先用 PLAN-38002 的实验验证"AI 能不能产出有用的分析"（自由文本模式）
2. **同时**用 code-survey 的选择题模式跑一个小实验（选 20 个 commit，用问卷分类），对比两种模式的准确率和审核成本
3. 根据结果决定哪个方向作为核心路径

长期：如果问卷模式效果更好，可以考虑把 ContextPack + Ask Agent 降级为辅助探索工具，把核心知识生产路径切换到问卷模式。
