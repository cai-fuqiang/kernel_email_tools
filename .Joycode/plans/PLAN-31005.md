# PLAN-31005: 统一信息工作台（Workspace）改造计划

> **Status**: draft
> **Updated**: 2026-05-07
> **Depends-on**: PLAN-31004（Phase 3 / 3.5 的卡片与 inspector 组件收尾）
> **Priority**: P2（等 31004 收尾后启动）

## 背景与动机

目前侧边栏 `Knowledge Workbench` 分组下并列 4 个独立页面：`Search Emails` / `Tags` / `Annotations` / `Translations`。它们在数据主体上高度重叠——本质都是围绕 email（以及未来的 code、manual、knowledge entity 等 target）展开，只是切入面不同：

- Search：按关键词/过滤器找 email thread
- Tags：按 tag 聚合 email / annotation / kernel line / knowledge entity
- Annotations：按批注找 email/code thread
- Translations：按翻译状态找 email thread

四个入口带来的代价：

1. **心智割裂**：tag → email、annotation → email、translation → thread 之间必须跨页跳转刷新。
2. **组件重复**：每个页面自己实现 list / filter / 结果卡，即使 PLAN-31004 在逐个做紧凑化，也是一份设计四处复制。
3. **侧边栏噪音**：4 个入口挤占有限的 nav 空间，新用户难以形成主干印象。

## 目标

- 将 `Search` / `Tags` / `Annotations` **三个视图**合并为一个 `/workspace` 页面（Translations 本轮不合并，见"范围裁剪"）。
- 建立统一的 `WorkspaceEntity` 抽象和 adapter 层，列表 UI 只有一套；详情面板按 kind 分发渲染。
- 不重写 PLAN-31004 产出的 `ResultCard` / `AnnotationCard` / `StickyContextBar` / `FloatingInspectorDock`，**只做外层壳**。
- 旧路由 `/search`、`/tags`、`/annotations` 保留并跳转到 `/workspace?view=...`，旧页面文件保留作为退路。

## 非目标 / 范围裁剪

- **不合并 Translations**。翻译是任务/进度视图（job 队列 + 缓存归档），和 entity 列表不同构。Translations 保留独立页面或作为 ThreadDrawer inspector dock（PLAN-31004 Phase 3.5 已落地）。
- **不改后端**。Stage 1 只做前端聚合，不新增 `/api/workspace/entities` 接口、不加迁移。Stage 2 是否需要后端统一接口，等 Stage 1 真实使用后再决定。
- **不推倒重来**。SearchPage / TagsPage / AnnotationsPage 文件保留；路由先做兼容跳转，物理删除要等到 Stage 1 上线 2 周以上无回归后。
- **不引入新框架**，不改权限模型，不改 API 契约。

## Stage 1 — 前端统一壳 + adapter（本 PLAN 主体）

### S1.1 路由与入口

- 新增路由 `/workspace`，默认视图 `view=email`。
- 侧边栏 `Knowledge Workbench` 分组：把 `Search Emails` / `Tags` / `Annotations` 三项替换为一个 `Workspace` 入口（图标 `LayoutGrid` 或 `Inbox`）。
- 保留 `Translations` 独立入口不动。
- 旧路由 `/search` / `/tags` / `/annotations` 保留页面实现，同时在 Workspace 的 nav 中提供"打开经典页面"入口；外链/书签继续可用。
- Dashboard / 其他页面中指向旧路由的链接全部改为 `/workspace?view=...`；旧 URL 仍能打开对应的经典页面。

### S1.2 统一实体模型

首期只收敛 **4 种 kind**，其余作为 Stage 2 扩展。

```ts
type WorkspaceEntityKind =
  | 'email_thread'      // email 搜索结果、tag 下的 email
  | 'annotation'        // 邮件批注 + 代码批注（统一）
  | 'tag'               // tag 本体（带 target 计数）
  | 'knowledge_entity'; // 可选：只读跳转，不编辑

type WorkspaceTarget = {
  type: 'email_thread' | 'email_message' | 'code_line' | 'sdm_section' | 'knowledge' | 'tag';
  ref: string;
  anchor?: { line_start?: number; line_end?: number; paragraph?: number };
};

type WorkspaceEntity = {
  id: string;                          // 全局唯一，形如 `email_thread:<thread_id>`
  kind: WorkspaceEntityKind;
  target: WorkspaceTarget;
  title: string;                       // 主行：subject / tag.name / annotation excerpt 首行
  subtitle?: string;                   // 次行：sender · date · channel / target label
  excerpt?: string;                    // 1 行摘要（line-clamp-1）
  badges: Array<{ label: string; tone?: 'muted'|'info'|'success'|'warning'|'danger' }>;
  meta: Array<{ label: string; value: string }>;  // 2–4 个 key-value
  counts?: Array<{ label: string; value: number }>; // tag.targets / thread.annotations 等
  updatedAt?: string;
  raw: unknown;                        // 原始后端对象，详情面板取用
};
```

**Adapter 约束**：每种 kind 对应一个纯函数 `toWorkspaceEntity(raw): WorkspaceEntity`，放在 `web/src/workspace/adapters/`，禁止在列表组件里做 kind 分支。

### S1.3 页面结构

```
┌──────────────────────────────────────────────────────────────┐
│ StickyContextBar: 当前 view + query + 命中数 + 主操作（复用）│
├──────────┬───────────────────────────────────┬───────────────┤
│ Left Nav │           Entity List              │  Detail Panel │
│ (可收起) │  紧凑卡片（复用 31004 ResultCard/  │  按 kind 分发 │
│          │   AnnotationCard 变体）            │  renderer     │
│ - view   │  选中项行内短展开（高度封顶）      │               │
│ - filter │  完整详情 → 右侧面板               │  - thread:    │
│ - tag    │                                    │    ThreadDrawer│
│   tree   │                                    │  - annotation:│
│          │                                    │    AnnotCard  │
│          │                                    │  - tag:       │
│          │                                    │    TagDetail  │
└──────────┴───────────────────────────────────┴───────────────┘
```

- **StickyContextBar**：直接复用 PLAN-31004 组件，增加 view 切换 dropdown。
- **列表卡片**：`email_thread` kind 用 `ResultCard`；`annotation` kind 用 `AnnotationCard` 的 grid 变体；`tag` kind 用 `TagSummaryCard`（新建，≤120 行）。**不重新发明**卡片内部结构。
- **详情面板**：
  - `email_thread` → 复用 `ThreadDrawer` 内嵌形态（非抽屉，直接占右栏）
  - `annotation` → 复用 `AnnotationCard` 完整形态 + "打开所属 thread" 按钮
  - `tag` → 展示 tag 元信息 + tag target 子列表（子列表本身还是 `WorkspaceEntity`，递归复用）
- **窄屏（< lg）**：Detail Panel 退化为抽屉；Left Nav 变底部 tabs。

### S1.4 数据聚合策略

Stage 1 纯前端聚合，每个 view 对应一组已有 API：

| view | API 调用 | adapter |
|------|----------|---------|
| `email` | `searchEmails` | `emailHitToEntity` |
| `tag` | `getTagTree` + 点开时 `getTagTargets` | `tagToEntity` / `tagTargetToEntity` |
| `annotation` | `listAnnotations` + `listCodeAnnotations` 合并 | `annotationToEntity` |

- 列表分页沿用各自 API 的分页参数，不做跨源合并分页（避免复杂度）。
- `q`（全局搜索）在 Stage 1 仅作用于当前 view，不做跨 view 联合搜索。
- 过滤器：kind（view 切换）、tag（复用 TagFilter）、target_type、status。

## Stage 2 — 观察期后再评估（不在本 PLAN 承诺）

Stage 1 上线后观察 2 周，收集以下信号再决定 Stage 2 范围：

- 是否需要 `GET /api/workspace/entities` 服务端统一接口（当 adapter 聚合分页/排序语义痛时再做）
- 是否把 Translations 并入（当发现翻译也频繁和 tag/search 混合使用时）
- 是否扩展 kind（`tag_target` 独立粒度、`translation_job`、`agent_run` 等）
- 是否物理删除 `SearchPage.tsx` / `TagsPage.tsx` / `AnnotationsPage.tsx`

Stage 2 启动时另立 PLAN，不在 31005 范围内。

- 如果 31004 落地过程中发现某些卡片 props 需要扩展（例如 `ResultCard` 支持不同 kind 图标），在 31004 里补充，不塞进 31005。

## 实施顺序

1. **前置**：确认 31004 Phase 3 / 3.5 的代码已合并（ResultCard / AnnotationCard grid / StickyContextBar / InspectorDock）。
2. **PR-1**：`web/src/workspace/types.ts` + 3 个 adapter 纯函数 + 单元测试。纯 TypeScript，无 UI。
3. **PR-2**：`EntityList` + `EntityDetailPanel` + `TagSummaryCard` 组件，Storybook/开发页下能独立渲染（用 mock adapter 数据）。
4. **PR-3**：`WorkspacePage.tsx` 接入三个真实 API view，路由注册 `/workspace`，侧边栏和 Dashboard 链接更新。旧路由保留。
5. **PR-4**：窄屏适配 + 移动端抽屉/tabs。

每个 PR 独立可合并、可回滚。

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| Adapter 退化成 kind-if-else | 代码评审强制：`EntityList` 中禁止出现 `entity.kind === ...` 分支；所有 kind 差异收敛在 adapter 和 detail renderer |
| 列表分页在多源合并时语义混乱 | Stage 1 明确：每个 view 只调一个主 API，不做跨源合并分页 |
| 旧页面用户习惯被破坏 | 旧路由保留 + 经典页面文件保留；侧边栏 Workspace 下提供"打开经典视图"入口 |
| 与 31004 的 inspector dock 冲突 | Workspace 右侧详情已常驻，inspector dock 在 Workspace 内不启用；仅旧路由/ThreadDrawer 单独打开时启用 |
| kind 过度扩张 | 首期锁死 4 种；新增 kind 必须先更新本 PLAN 或新开 PLAN |

## 验收标准

- [ ] `/workspace?view=email` 能完成原 SearchPage 的主流程：搜索、过滤、打开 thread、批量打 tag、summarize draft。
- [ ] `/workspace?view=tag` 能完成原 TagsPage 的主流程：看 tag 树、看 tag 下的 targets、打开 email thread。
- [ ] `/workspace?view=annotation` 能完成原 AnnotationsPage 的主流程：搜索、按类型/状态过滤、打开批注所在 target、发布审核。
- [ ] `/search`、`/tags`、`/annotations` 旧 URL 仍可打开对应经典页面，不报错。
- [ ] Dashboard 中的 Quick action 跳转到 `/workspace?view=...`，进入后 view 正确选中。
- [ ] `EntityList` 组件内没有 `entity.kind === 'xxx'` 分支（adapter + renderer map 除外）。
- [ ] 窄屏（≤ md）：Left Nav 变 tabs，Detail Panel 变抽屉，无横向滚动。
- [ ] `npm run lint` + `npm run build` 通过。
- [ ] 手工冒烟：搜索、打开 thread、tag target 跳转、annotation 跳转、发布审核均无回归。
