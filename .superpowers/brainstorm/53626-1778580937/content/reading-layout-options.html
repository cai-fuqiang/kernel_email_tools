阅读/编辑优先的三个布局方案

请看三个方向的空间分配。我的推荐是 A：把实体详情做成“文章式编辑台”，同时保留证据和关系的快速侧栏入口。

Navigator
Context Rail

A. 文章式编辑台（推荐）

左侧列表变窄，只负责找实体；中间是宽阅读/编辑面板；右侧常驻上下文栏放标签、状态、关系摘要、证据入口和保存动作。

优点

  • 阅读宽度稳定
  • 编辑动作不迷路
  • 和现有组件改动较吻合

代价

  • 小屏需要把右栏折叠成抽屉
Navigator

B. Tabbed 编辑器

把 Overview、Explanation、Relations、Evidence、History 都变成页签，减少长滚动，让用户只看当前任务。

优点

  • 页面更短
  • 任务切换清楚
  • 适合单点编辑

代价

  • 证据和关系容易被隐藏
  • 跨区对照成本更高
Search + Queue

C. 文档阅读器

把知识库当成文档集合，主区域几乎全给阅读内容；搜索、实体队列、草稿和元信息全部放到可展开右栏。

优点

  • 阅读最沉浸
  • 适合长解释
  • 视觉最安静

代价

  • 实体切换变慢
  • 管理型操作入口更深
